## 5. 프로젝트 평가 및 개선

이 섹션은 프로젝트 본편(2025.09-2025.11) 당시 남아 있던 문제와 개선 방향, 그리고 2026.03에 진행한 저장소 기준의 후속 하드닝을 함께 정리합니다. 먼저 본편 시점의 한계를 정리하고, 이어서 실제 저장소 기준으로 어떤 보강을 했는지 덧붙였습니다.

~~~troubleshooting
제목: 도메인 중심 API와 화면 요구사항의 불일치

문제:
- 각 도메인(주문, 견적 등) API는 존재했지만, 대시보드나 운영 화면에서 필요한 정렬·검색·페이지네이션을 즉각 제공하지 못했습니다.
- 특히 ERP 특성상 단건 CRUD보다 다량의 상태를 빠르게 파악하는 조회가 핵심임에도, 조회 계약이 느슨하여 프런트엔드 비효율이 발생했습니다.

원인:
- 초기 개발이 기능 단위 API 구축에 매몰되어, 복합적인 조건을 요구하는 대시보드 시나리오가 API 계약 단계에서 고려되지 않았습니다.

해결:
- 견적/통계 API에 필터, 정렬, 페이지네이션 파라미터를 통합하여 화면 요구사항에 맞춤화된 전문 조회 계약을 정립했습니다.
- 주문/견적 대시보드 전용 서비스를 구축하여 제품 정보를 결합한 UI 최적화 모델을 제공하도록 개선했습니다.

결과:
조회 API를 단순 기능의 부속물이 아닌 독립적인 운영 계약으로 재정의하였으며, 이후 화면 요구사항을 기반으로 조회 모델을 역설계하는 프로세스를 구축했습니다.
~~~

~~~troubleshooting
제목: 인증과 인가 레이어의 책임 혼재

문제:
- 로그인 성공 직후 사용자가 자신의 역할(내부자, 고객사, 공급사)에 맞지 않는 클라이언트나 API에 접근할 수 있는 보안 허점이 있었습니다.
- 권한 해석 기준이 명확하지 않아 ERP 전체의 보안 경계가 모호해지는 문제가 발생했습니다.

원인:
- 프로젝트 초기 인증(Authentication)과 인가(Authorization)를 단일 흐름으로 파악하여, Auth 서버와 Gateway 간의 책임 분리가 이루어지지 않았습니다.

해결:
- Auth 서버에서 모바일 클라이언트 접근 제어를 강화하고, Gateway에서는 모듈별 API에 역할 기반 권한 검사를 강제하도록 구조를 분리했습니다.

결과:
인증, 라우팅, 권한 해석의 주체를 명확히 분리함으로써, 신규 도메인 추가 시에도 일관된 보안 정책을 즉각 적용할 수 있는 기반을 마련했습니다.
~~~

~~~troubleshooting
제목: 비동기 분산 환경의 데이터 정합성 결여

문제:
- 공급사 생성이나 내부 사용자 생성 시 여러 서비스의 데이터가 함께 성공해야 하지만, 부분 실패 시 불완전한 데이터가 남는 현상이 있었습니다.
- 서비스 간 통신 과정에서 이벤트가 누락되거나 결과 상태가 모호하여 운영 안정성이 저하되었습니다.

원인:
- `@Transactional`만으로 해결할 수 없는 서비스 간 비동기 흐름에 대한 전역 트랜잭션 정책(Saga 패턴 등)이 부재했습니다.

해결:
- `SagaTransactionManager`를 도입하여 트랜잭션 ID 단위로 변경 내역을 관리하고, 실패 시 자동 보상 로직이 실행되도록 설계했습니다.
- Kafka 연계 흐름에 수동 Ack와 결과 확인 모듈을 추가하여 비동기 작업 상태를 추적할 수 있는 구조를 마련했습니다.

결과:
로컬 트랜잭션과 분산 트랜잭션을 분리해 사고하는 관점을 확보했고, 보상 기반 처리 구조의 필요성을 실제 코드로 검증했습니다. 다만 메시지 재시도 정책, DLQ, 발행-저장 원자성은 프로젝트 본편 이후에도 남아 있던 보강 과제였습니다.
~~~

### 후속 개선 (2026.03)

아래 내용은 프로젝트 본편 종료 후, 현재 저장소 기준으로 진행한 후속 하드닝입니다. 본편에서 남아 있던 "계약 정리", "Gateway 운영성", "브라우저 인증 디테일"을 보강하는 작업에 가깝습니다.

~~~troubleshooting
제목: 응답 계약과 오류 포맷의 후속 표준화

문제:
- 서비스별로 `ApiResponse` wrapper와 raw DTO가 섞여 있어 프런트엔드와 Gateway의 응답 해석 비용이 컸습니다.
- 앱마다 오류 응답 형식도 조금씩 달라 후속 리팩토링 범위가 넓어지는 문제가 있었습니다.

원인:
- 프로젝트 본편에서는 기능 확장과 도메인 연계가 우선이었고, 성공/실패 응답의 전역 규약 정리는 뒤로 밀려 있었습니다.

해결:
- Auth, Business, SCM, Alarm 계열의 오류 응답을 `ProblemDetail` 기준으로 맞추고, 성공 응답은 wrapper를 줄여 raw DTO 중심으로 정리했습니다.
- 비동기 `DeferredResult` 결과 관리자도 wrapper 없는 응답 계약에 맞춰 수정하고, 프런트엔드 API도 raw 응답 기준으로 함께 정렬했습니다.

```chips
auth mixed-mode 응답 계약 정리 | https://github.com/Gumraze-git/everp/commit/e687f9072b5477aa05a4d9cbb3749f00e1a2ba67 | commit
provider 응답 계약 정리 | https://github.com/Gumraze-git/everp/commit/857987567cd7c3306617e7ec596344a0e26bf074 | commit
frontend raw 응답 계약 정리 | https://github.com/Gumraze-git/everp/commit/a8dc846a6b1a10b4d013c9246e38bb2235a4770a | commit
```

결과:
"오류 응답은 표준화하고 성공 응답은 단순화한다"는 기준을 저장소 전반에 적용했습니다. 프로젝트 본편의 기능 구현을 운영 가능한 계약 수준으로 정리한 후속 작업이었습니다.
~~~

~~~troubleshooting
제목: Gateway 외부 호출과 문서 레이어의 정리

문제:
- Gateway의 외부 호출 로직이 `WebClient`와 개별 구현 스타일로 흩어져 있어 Blocking 호출 기준으로 보면 코드 가독성과 유지보수성이 떨어졌습니다.
- Swagger 문서화 어노테이션도 컨트롤러 내부에 강하게 결합되어 구현과 문서를 함께 수정해야 하는 부담이 컸습니다.

원인:
- 프로젝트 본편에서는 기능 전달과 라우팅 완성이 우선이었고, 공통 호출 클라이언트와 문서 계약 레이어를 뒤늦게 정리하게 되었습니다.

해결:
- Gateway의 Blocking HTTP 호출을 `RestClient` 기반으로 일원화하고, 예외 릴레이도 공통 방식으로 정리했습니다.
- Swagger 어노테이션은 `*Api` 인터페이스로 분리해 컨트롤러는 구현 책임에 집중하도록 구조를 나눴습니다.

```chips
blocking http client를 restclient로 전환 | https://github.com/Gumraze-git/everp/commit/4871bfc99d1a41bb4b508839ddf7f00fd87acbbd | commit
스웨거 문서 인터페이스로 분리 | https://github.com/Gumraze-git/everp/commit/c2e1c214b6de9bbd0b1acee1c524e269f1d5a21b | commit
```

결과:
Gateway 외부 연동과 문서화 코드를 계약 중심 구조로 다시 정리하면서, 본편에서 만든 경계를 유지보수 가능한 형태로 다듬었습니다.
~~~

~~~troubleshooting
제목: 쿠키 기반 인증과 공개 경로의 운영 하드닝

문제:
- 프로젝트 본편 이후 브라우저 환경에서 인증 bootstrap, CSRF, 로그인 후 복귀 흐름을 더 안정적으로 맞출 필요가 있었습니다.
- GW와 각 서비스의 공개 경로도 모듈별로 세부 규약이 조금씩 달라 운영 중 해석 비용이 남아 있었습니다.

원인:
- 본편 시점에는 기능 흐름과 권한 경계를 우선 맞췄고, 웹 전용 인증 디테일과 resource-oriented 경로 정리는 후순위였습니다.

해결:
- 쿠키 기반 토큰 교환, CSRF bootstrap, 로그인 성공 후 프런트 복귀 흐름을 정리했습니다.
- SD, PP, IM 등 공개 경로를 리소스 중심 규약으로 다시 맞추고 GW relay 경로도 함께 정리했습니다.

```chips
쿠키 기반 GW 인증과 CSRF 부트스트랩 정리 | https://github.com/Gumraze-git/everp/commit/eb6a446772bdedca3817122fe8d4122adbdb6e64 | commit
직접 로그인 성공 후 프론트로 복귀 | https://github.com/Gumraze-git/everp/commit/1a3d36a194ef8029267279fdc3591b8712e5fa7e | commit
sd 공개 api와 gw 경로 정렬 | https://github.com/Gumraze-git/everp/commit/32bd5f532a99bbcb7b862ded216c38444f8d82ef | commit
im lifecycle와 헬퍼 경로 정리 | https://github.com/Gumraze-git/everp/commit/c2647e890a7ef533d46e51ee33e0c4aed7260ecd | commit
```

결과:
프로젝트 본편에서 만든 인증 경계와 API 계약을 실제 사용 흐름에 맞게 다시 다듬으면서, FE-GW-Backend 간 해석 차이를 줄이는 후속 하드닝을 진행했습니다.
~~~
