## 5. 프로젝트 평가 및 개선

~~~troubleshooting
제목: 도메인 중심 API와 화면 요구사항의 불일치

문제:
- 각 도메인(주문, 견적 등) API는 존재했지만, 대시보드나 운영 화면에서 필요한 정렬·검색·페이지네이션을 즉각 제공하지 못했습니다.
- 특히 ERP 특성상 단건 CRUD보다 다량의 상태를 빠르게 파악하는 조회가 핵심임에도, 조회 계약이 느슨하여 프런트엔드 비효율이 발생했습니다.

원인:
- 초기 개발이 기능 단위 API 구축에 매몰되어, 복합적인 조건을 요구하는 대시보드 시나리오가 API 계약 단계에서 고려되지 않았습니다.

해결:
- 견적/통계 API에 필터, 정렬, 페이지네이션 파라미터를 통합하여 화면 요구사항에 맞춤화된 전문 조회 계약을 정립했습니다.
- 주문/견적 대시보드 전용 서비스를 구축하여 제품 정보를 결합한 UI 최적화 모델을 제공하도록 개선했습니다.

결과:
조회 API를 단순 기능의 부속물이 아닌 독립적인 운영 계약으로 재정의하였으며, 이후 화면 요구사항을 기반으로 조회 모델을 역설계하는 프로세스를 구축했습니다.
~~~

~~~troubleshooting
제목: 인증과 인가 레이어의 책임 혼재

문제:
- 로그인 성공 직후 사용자가 자신의 역할(내부자, 고객사, 공급사)에 맞지 않는 클라이언트나 API에 접근할 수 있는 보안 허점이 있었습니다.
- 권한 해석 기준이 명확하지 않아 ERP 전체의 보안 경계가 모호해지는 문제가 발생했습니다.

원인:
- 프로젝트 초기 인증(Authentication)과 인가(Authorization)를 단일 흐름으로 파악하여, Auth 서버와 Gateway 간의 책임 분리가 이루어지지 않았습니다.

해결:
- Auth 서버에서 모바일 클라이언트 접근 제어를 강화하고, Gateway에서는 모듈별 API에 역할 기반 권한 검사를 강제하도록 구조를 분리했습니다.

결과:
인증, 라우팅, 권한 해석의 주체를 명확히 분리함으로써, 신규 도메인 추가 시에도 일관된 보안 정책을 즉각 적용할 수 있는 기반을 마련했습니다.
~~~

~~~troubleshooting
제목: 비동기 분산 환경의 데이터 정합성 결여

문제:
- 공급사 생성이나 내부 사용자 생성 시 여러 서비스의 데이터가 함께 성공해야 하지만, 부분 실패 시 불완전한 데이터가 남는 현상이 있었습니다.
- 서비스 간 통신 과정에서 이벤트가 누락되거나 결과 상태가 모호하여 운영 안정성이 저하되었습니다.

원인:
- `@Transactional`만으로 해결할 수 없는 서비스 간 비동기 흐름에 대한 전역 트랜잭션 정책(Saga 패턴 등)이 부재했습니다.

해결:
- `SagaTransactionManager`를 도입하여 트랜잭션 ID 단위로 변경 내역을 관리하고, 실패 시 자동 보상 로직이 실행되도록 설계했습니다.
- Kafka 연계 흐름에 수동 Ack와 결과 확인 모듈을 추가하여 비동기 작업의 완결성을 확보했습니다.

결과:
로컬 트랜잭션과 분산 트랜잭션을 분리하여 설계하는 관점을 확보하였으며, 실패를 전제로 한 안정적인 비동기 처리 구조를 정립했습니다.
~~~

~~~troubleshooting
제목: 파편화된 API 응답 규약 및 오류 포맷

문제:
- 각 앱마다 `ApiResponse` wrapper, fallback 응답 등이 혼재되어 프런트엔드의 응답 처리 비용이 과다하게 발생했습니다.
- Gateway, Auth, Provider 계층마다 에러 응답 형식이 달라 리팩토링 및 유지보수에 어려움이 있었습니다.

원인:
- 기능 개발 속도를 우선시하면서 서비스별 응답 형식이 부분적으로 파편화되었고, 공통 규약에 대한 강제가 이루어지지 않았습니다.

해결:
- 모노레포 정리 단계에서 모든 응답 계약을 `ProblemDetail` 표준으로 통일했습니다.
- 불필요한 wrapper 반환을 제거하고 DTO와 공참 예외 처리기를 통해 응답 구조를 단일화했습니다.

결과:
"응답 계약은 서비스와 무관하게 전역적으로 동일해야 한다"는 기준을 세웠으며, 이를 통해 API 해석 오차를 제거하고 개발 생산성을 향상시켰습니다.
~~~

~~~troubleshooting
제목: Gateway 외부 호출 구현의 복잡성

문제:
- Gateway의 외부 API 호출 로직이 `WebClient` 기반의 비정형적인 스타일로 흩어져 있어 관리 효율이 낮았습니다.
- Blocking 호출임에도 Reactive 스타일이 혼용되어 코드 가독성과 유지보수성이 저하되는 문제가 있었습니다.

원인:
- 초기 개발 시 각 클라이언트(alarm, business 등)가 별도 스타일로 구현되었고, 공통 호출 인터페이스가 부재했습니다.

해결:
- `RestClientProvider`를 도입하여 모든 Blocking 호출을 `RestClient` 기반으로 일원화했습니다.
- 예외 처리 방식을 `ProblemDetailFactory`로 통합하여 일관된 에러 릴레이 구조를 구축했습니다.

결과:
Gateway의 외부 연동 구조를 단순화하고 호출 방식을 공통화함으로써, 신규 외부 서비스 연계 시의 개발 속도와 코드 품질을 확보했습니다.
~~~

~~~troubleshooting
제목: 컨트롤러와 문서화 코드의 강한 결합

문제:
- 컨트롤러 내부에 Swagger 어노테이션이 산재하여 비즈니스 로직 파악이 어렵고, 코드 변경 시 문서가 동기화되지 않는 리스크가 있었습니다.
- 공통 보안 및 에러 정의가 반복 작성되어 문서 데이터의 중복과 파편화가 발생했습니다.

원인:
- 기능 구현 중심의 개발로 인해 문서화 책임이 컨트롤러 내부로 몰렸고, 전역적인 문서 규약 관리 레이어가 없었습니다.

해결:
- Swagger 어노테이션을 `*Api` 인터페이스로 완전히 격리하고 컨트롤러는 이를 구현만 하도록 책임을 분리했습니다.
- 공통 보안 요구사항과 에러 응답 메타 어노테이션을 자동 적용하도록 커스터마이징했습니다.

결과:
문서를 기술적 계약(Contract) 레이어로 승격시켜 구현부의 가독성을 높였으며, 문서의 정확성과 신뢰도를 동시에 확보했습니다.
~~~
