## 5. 프로젝트 평가

Everp는 화면 수나 기능 수보다, **복합 업무 도메인을 어떤 경계와 계약으로 설명 가능하게 만들었는가**를 보여주는 프로젝트였습니다. 이 프로젝트의 가치는 "ERP를 만들었다"보다, 공통 인증, Gateway 권한 처리, Problem Detail 기반 응답 계약, 조회 계약, 초기화 데이터 기준을 팀이 같은 언어로 사용할 수 있게 묶어낸 경험에 있습니다.

### 이 프로젝트가 강하게 보여주는 점

- **공통 인증과 권한 해석 경계를 먼저 고정한 점**
  - Auth, Gateway, Business, SCM, Alarm의 역할을 나누고, 특히 Gateway를 단순 라우터가 아니라 권한 해석 지점으로 고정한 점이 분명합니다.
  - 덕분에 인증 성공과 업무 접근 허용을 별개의 문제로 다룰 수 있었고, 이후 도메인 API를 늘려도 해석 기준을 유지할 수 있었습니다.

- **Bounded Context와 공통 언어로 경계를 설명한 점**
  - Business와 SCM를 구현 단위보다 먼저 설명 가능한 업무 경계로 두고, 각 컨텍스트가 어떤 데이터와 조회 계약을 공유하는지 정리했습니다.
  - 이는 ERP를 단순 CRUD 모듈 모음이 아니라 운영 가능한 업무 구조로 본 관점에 가깝습니다.

- **응답 계약과 조회 계약을 먼저 정리한 점**
  - Problem Detail 기반 오류 응답과 대시보드/목록용 조회 계약을 정리해 프런트와 백엔드가 같은 기준으로 화면을 연결할 수 있게 했습니다.
  - 즉, API를 구현의 부산물이 아니라 운영 계약으로 다루는 관점이 살아 있습니다.

- **팀이 같은 규칙으로 개발할 수 있게 만든 점**
  - 초기화 데이터와 테스트 계정 규칙을 정리해, 각 모듈이 같은 시나리오와 같은 권한 기준 위에서 개발될 수 있게 만들었습니다.
  - 팀 프로젝트에서 이 기준을 앞쪽에 고정한 경험이 이 프로젝트의 중요한 가치였습니다.

### 포트폴리오 관점에서 보완해서 읽어야 할 점

- **구조 강점에 비해 사용자 가치 설명은 더 보강할 수 있습니다**
  - 이 프로젝트는 구조와 계약 설명은 강하지만, 최종 사용자가 어떤 업무를 더 안정적으로 처리하게 되었는지에 대한 묘사는 상대적으로 적습니다.
  - 그래서 포트폴리오나 devlog에서는 기술 복잡도보다 "왜 이 기준이 운영과 협업에 유리했는가"를 함께 묶어 설명하는 편이 더 효과적입니다.

- **분산 시스템 완성도보다 우선순위 판단이 핵심입니다**
  - 비동기 연계와 Saga는 분명 중요한 구현 근거지만, 이 프로젝트의 주된 가치는 `공통 인증/Gateway/응답 계약/조회 계약`을 먼저 고정한 판단에 있습니다.
  - 따라서 devlog에서도 분산 시스템 자체를 과장하기보다, 어떤 기준을 먼저 합의하고 코드로 옮겼는지에 초점을 맞추는 편이 적절합니다.

- **팀 프로젝트 특성상 개인 기여를 계속 선명하게 유지해야 합니다**
  - 공통 인증 서버, Gateway 권한 처리, 응답 계약, 조회 계약, 초기화 데이터처럼 본인 기여 축은 강하지만, 정보량이 많아지면 읽는 사람이 핵심 기여를 놓칠 수 있습니다.
  - 그래서 현재처럼 역할과 설계 판단을 앞쪽 섹션에서 먼저 고정해 주는 방식이 중요합니다.

### 종합 평가

Everp는 "많은 기능을 구현한 프로젝트"라기보다, **복잡한 업무 도메인을 어떤 기준으로 나누고 팀이 같은 규칙으로 개발할 수 있게 만들었는가**를 보여주는 프로젝트로 평가하는 것이 가장 적절합니다. 포트폴리오에서도 이 프로젝트는 대규모 분산 시스템을 완성했다는 식으로 과장하기보다, 제한된 기간과 팀 규모 안에서 가장 중요한 공통 인증·권한 경계·응답 계약·조회 계약을 먼저 고정한 경험으로 설명할 때 가장 설득력이 높습니다.
