## 1. 프로젝트 개요 및 목표

**RallyOn**은 배드민턴 자유게임 운영을 더 안정적으로 관리할 수 있게 만든 서비스입니다.

### 배경 및 목적

RallyOn에서 먼저 풀어야 했던 문제는 화면 수보다 **실행 경계와 운영 규칙**이었습니다. 배드민턴 모임 운영은 참가자 등록, 라운드 편성, 코트 배정, 운영자 수정 권한이 한 흐름으로 이어지기 때문에, UI만 먼저 늘리면 규칙이 쉽게 흔들립니다.

그래서 저는 기능 소개보다 먼저 아래 세 가지를 고정했습니다.

- **브라우저 인증 경계**: secure cookie를 쓰는 이상 `localhost` 포트 조합만으로는 실제 동작을 충분히 검증하기 어렵기 때문에, 로컬 HTTPS와 프록시를 포함한 실행 환경부터 맞췄습니다.
- **운영 규칙 경계**: organizer만 수정할 수 있는 운영 API와 외부 공개용 share 조회 API를 분리하고, 같은 라운드 안의 중복 배정 같은 규칙을 서버에서 검증했습니다.
- **온보딩 계약 정리**: 로그인 후 프로필 작성, 지역 선택, 장소 검색, 자유게임 생성이 서로 어긋나지 않도록 프로필/지역/장소 계약을 정리했습니다.

이 기준을 먼저 고정해 두면 프런트는 로그인 UI와 사용자 흐름에 집중하고, 서버는 인증/운영 규칙과 데이터 접근 제어를 담당하는 구조를 유지할 수 있습니다.

### 핵심 사용자 시나리오

```mermaid
flowchart LR
  A["로그인"] --> B["자유게임 생성"]
  B --> C["게임 이름/날짜/장소 설정"]
  C --> D["참가자 등록 / 게임 편성"]
  D --> E["운영자 전용 게임 관리"]
  E --> F["공유 코드로 게임 조회"]
```

- 로그인한 사용자는 자유게임을 생성하고 날짜, 장소, 코트 수 같은 운영 설정을 저장할 수 있습니다.
- 운영자(`organizer`)는 참가자 등록과 라운드/매치 편성을 관리할 수 있으며, 수정 권한은 운영자에게만 부여됩니다.
- 공유 코드는 외부 공개 조회에 사용할 수 있고, 운영 상세 조회와 수정은 인증 경로로 분리됩니다.

### 주요 기능

```features
- **브라우저 인증 흐름 구현**: OAuth 2.1/OIDC Authorization Code + PKCE, secure cookie, 토큰 재발급 흐름을 기준으로 브라우저 인증 경계를 구현했습니다. 인증 화면 전환은 프런트가 담당하고, 인가 코드 교환·토큰 검증·세션 재발급은 서버가 담당하도록 역할을 나눴습니다.

- **운영 규칙 중심 자유게임 API**: 로그인한 사용자는 자유게임 세션을 생성하고 날짜, 장소, 코트 수 같은 운영 설정을 저장할 수 있습니다. 운영자는 참가자 등록과 라운드/매치 편성을 관리할 수 있고, 수정 권한은 organizer에게만 부여됩니다.

- **공개 조회와 운영 API 분리**: shareCode 기반 공개 조회는 외부 사용자가 읽을 수 있도록 열어두고, 운영 상세 조회와 수정은 인증 경로로 분리해 운영 규칙이 공개 조회 경로로 섞이지 않도록 했습니다.

- **계약 안정화와 검증**: 지역 선택, 장소 검색, 자유게임 생성 validation을 계약으로 정리하고, 기능 테스트와 ArchUnit 규칙으로 인증/운영 구조가 유지되는지 함께 확인했습니다.
```
