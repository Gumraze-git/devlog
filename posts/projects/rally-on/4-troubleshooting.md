## 4. 핵심 트러블슈팅

~~~troubleshooting
제목: DDD와 헥사고날 아키텍처를 통한 도메인 경계 분리

문제:
- 레이어 구조 안에서 인증, 사용자, 자유게임, 외부 의존이 한 흐름처럼 붙어 있으면 도메인 경계가 흐려지고, 자유게임 운영 규칙이 인프라나 웹 계층의 구현 세부에 쉽게 섞였습니다.
- 특히 세션 생성, 설정 변경, 참가자 관리, 라운드/매치 편성이 하나의 거대한 서비스 흐름으로 묶이면 변경 단위를 설명하기도 검증하기도 어려웠습니다.

원인:
- 레이어 구조만으로 기능을 쌓으면 사용자, 자유게임, 외부 의존이 같은 서비스 계층에 머물기 쉬워 도메인 변경 경계가 흐려집니다.
- 운영 단위별 use case를 먼저 나누지 않으면 구조를 바꾸는 순간 비즈니스 규칙과 외부 연동 코드가 같이 흔들립니다.

해결:
- port-in / app service / domain / port-out / adapter-out 구조로 계층을 재정리했습니다.
- 자유게임 세션 본체, 설정, 참가자, 라운드/매치 편성, 공개 조회를 운영 단위 기준으로 나누고, 각 use case가 다루는 변경 경계를 분명하게 했습니다.
- 외부 OAuth, 장소 검색 같은 연동은 adapter-out으로 밀어 도메인 규칙과 분리했습니다.

> !compare
> ```chips
> As-Is | | asis
> ```
> ```text
> controller -> service -> repository 안에
> 인증 / 사용자 / 자유게임 / 외부 의존이 함께 섞임
> ```
>
> ```chips
> To-Be | | tobe
> ```
> ```text
> port-in / app service / domain / port-out / adapter-out
> + 자유게임 운영 단위별 use case 분리
> + 외부 의존 adapter-out 분리
> ```

결과:
자유게임과 사용자 도메인의 변경 경계를 분리하고, 운영 단위를 구조 안에서 설명 가능하게 고정할 수 있었습니다.
~~~

~~~troubleshooting
제목: TDD와 ArchUnit 기반 구조 검증

문제:
- 인증/세션/토큰/운영 규칙 변경이 누적되면, 기능은 겉으로 동작하더라도 구조와 경계가 함께 무너질 위험이 있었습니다.
- 특히 로그인, 세션 재발급, organizer 권한, 공개 조회 같은 흐름은 단순 API 응답만 확인해서는 회귀를 잡기 어려웠습니다.

원인:
- 인증 흐름과 자유게임 구조를 계속 손보는 상황에서 테스트가 기능 확인에만 머물면, 모듈 의존 방향과 운영 규칙 경계가 깨져도 늦게 발견됩니다.
- 요청 validation, 인증 시나리오, use case 규칙, 모듈 구조 검증이 서로 다른 층위인데 이를 함께 확인할 기준이 없으면 구조 변경이 불안정해집니다.

해결:
- use-case 테스트로 organizer 권한, 참가자 배정, 공개 조회 같은 운영 규칙을 검증했습니다.
- controller validation 테스트로 요청 필드 오류와 인증/인가 실패를 분리해 확인했습니다.
- 인증 시나리오 테스트와 ArchUnit 규칙을 함께 도입해 기능 동작과 모듈 의존 방향을 동시에 검증했습니다.

> !compare
> ```chips
> As-Is | | asis
> ```
> ```text
> 기능이 동작하는지만 수동으로 확인
> -> 구조 변경 시 인증 경계와 모듈 의존이 늦게 깨짐
> ```
>
> ```chips
> To-Be | | tobe
> ```
> ```text
> use-case 테스트
> + controller validation 테스트
> + 인증 시나리오 테스트
> + ArchUnit 의존 방향 규칙
> ```

결과:
기능 동작과 모듈 경계를 함께 검증하는 기준을 확보해, 구조를 바꿀 때도 인증 흐름과 운영 규칙을 같은 축에서 회귀 확인할 수 있게 됐습니다.
~~~

~~~troubleshooting
제목: OAuth 2.1 기반 인증·인가 구조 재설계

문제:
- 로컬 로그인 도입 이후 인증 책임이 서비스 전반으로 확산되면서, 계정·세션·토큰 책임이 한 레이어에 붙기 시작했습니다.
- 브라우저에서 보이는 로그인 흐름과 비즈니스 도메인 접근 규칙이 같은 서비스 구조 안에 섞이면 인증 변경이 자유게임 도메인까지 직접 영향을 줄 수 있었습니다.

원인:
- 인증 성공과 리소스 접근 허용을 별도 문제로 나누지 않으면, 세션/토큰 처리 책임이 비즈니스 로직과 같은 위치에 머물기 쉽습니다.
- OAuth 2.1/OIDC, Authorization Code + PKCE, 세션 재발급 흐름을 구조로 설명할 기준이 없으면 인증 로직이 점점 서비스 전반으로 퍼집니다.

해결:
- 단일 애플리케이션 안에서도 Authorization 역할과 Resource 역할을 분리해 설명 가능한 구조로 재정리했습니다.
- 로그인 화면 전환은 프런트가 담당하고, 인가 코드 교환·토큰 검증·세션 재발급은 서버가 담당하도록 역할을 나눴습니다.
- 이 구조를 기준으로 OAuth 2.1/OIDC, Authorization Code + PKCE, access/refresh 토큰 흐름을 정리했습니다.

> !compare
> ```chips
> As-Is | | asis
> ```
> ```text
> 로컬 로그인 이후
> 계정 / 세션 / 토큰 책임이 한 레이어에 붙음
> -> 인증 변경이 도메인 구조까지 직접 흔듦
> ```
>
> ```chips
> To-Be | | tobe
> ```
> ```text
> Authorization 역할 분리
> + Resource 역할 분리
> + OAuth 2.1 / OIDC / PKCE 흐름 기준 정리
> ```

결과:
인증 변경이 비즈니스 도메인까지 번지지 않도록 경계를 고정하고, 인증·인가 구조를 한 단계 높은 수준에서 설명 가능한 형태로 정리할 수 있었습니다.
~~~

~~~troubleshooting
제목: reverse proxy 기반 브라우저 인증 흐름 설계

문제:
- 브라우저에서 인증 화면, 토큰 처리, 운영 API 요청이 한 오리진에 섞여 있으면 secure cookie 기반 흐름을 실제 조건에서 검증하기 어려웠습니다.
- 단순 `http://localhost` 포트 조합만으로는 redirect URI, 쿠키 정책, refresh 흐름을 브라우저 기준으로 재현하기 어려웠습니다.

원인:
- secure cookie 인증은 백엔드 코드만으로 끝나지 않고, 브라우저의 쿠키 정책과 HTTPS 조건, host 분리에 직접 영향을 받습니다.
- 인증 책임을 구조적으로 나눠도, 실행 환경이 이를 검증하지 못하면 브라우저 인증은 계속 추상적인 설계에 머물게 됩니다.

해결:
- `auth.rallyon.test` / `api.rallyon.test` 호스트를 분리하고 nginx reverse proxy를 두어 브라우저 기준 실행 경계를 확정했습니다.
- `mkcert`, `.test` 도메인, DUMMY provider, refresh 흐름을 기준으로 로컬에서도 secure cookie 인증을 반복 검증할 수 있게 했습니다.
- `make up-live fe` 워크플로를 추가해 브라우저 인증 조건을 유지한 채 프런트 개발 속도도 확보했습니다.

> !compare
> ```chips
> As-Is | | asis
> ```
> ```text
> http://localhost:3000 + 별도 백엔드 포트
> -> secure cookie와 redirect 흐름을 매번 우회
> ```
>
> ```chips
> To-Be | | tobe
> ```
> ```text
> https://auth.rallyon.test / https://api.rallyon.test
> + nginx reverse proxy
> + mkcert
> + DUMMY login
> -> 브라우저 기준 인증 흐름을 로컬에서도 반복 검증
> ```

결과:
프런트는 로그인 UI를, 서버는 코드 교환·토큰 검증·재발급을 담당하는 구조를 브라우저 기준으로 반복 검증할 수 있게 됐습니다.
~~~
